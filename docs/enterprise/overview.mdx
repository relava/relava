# Enterprise

<Warning>
  All features on this page are **planned / coming soon** unless marked otherwise. The current release supports the Open Source tier (local CLI + server) only.
</Warning>

Relava's architecture is designed for enterprise deployment. The local-first MVP uses the same REST API, storage traits, and CLI interface that enterprise features build on.

## Deployment Tiers

| Tier | Description |
|------|-------------|
| **Open Source** | CLI + local server for individual developers. Free, runs on your machine. ✅ Available |
| **Cloud Hosted** | Relava-hosted registry for teams. Shared resource registry with auth. 🔜 Planned |
| **Self-Hosted** | On-premise registry for enterprises. Full control over data and access. 🔜 Planned |

Switching between tiers is a URL change — the CLI is the same everywhere.

## Scoping & Namespaces <sup>Planned</sup>

Resources support three scope levels:

| Scope | Format | Visibility |
|-------|--------|------------|
| **Global** | `code-review` | All users (current MVP behavior) |
| **Team** | `@platform-team/deploy-check` | Team members only |
| **Personal** | `@alice/code-review` | Owner only (shareable) |

Permissions are additive — a user's effective access is the union of their personal, team, and global permissions. Team admins can share team resources with other teams or individuals without moving them to global scope.

## Authentication & SSO <sup>Planned</sup>

Authentication is planned in progressive tiers:

1. **API Tokens** — server-generated bearer tokens, stored in `~/.relava/config.toml`. Works for CI/CD.
2. **Username/Password** — `relava login` command with locally-stored tokens.
3. **OIDC/SSO** — integration with Okta, Azure AD, Google Workspace via OpenID Connect. Browser-based OAuth flow.
4. **SAML** — for organizations that require SAML-based single sign-on.

The API already accepts `Authorization: Bearer <token>` headers — ignored in the local MVP, enforced when auth is enabled.

## Registry Federation <sup>Planned</sup>

Enterprise projects often need resources from multiple registries. The manifest supports multi-registry resolution:

```toml
agent_type = "claude"

[registries]
urls = [
  "https://registry.company.com",
  "http://localhost:7420",
]
```

Resolution is top-down: the first registry that has the requested resource wins. This lets company registries override public resources. `relava publish` targets the first URL by default.

## Storage Abstraction <sup>Planned</sup>

The server is built on storage abstraction traits that allow swapping backends without changing business logic:

| Layer | Local (MVP) | Enterprise |
|-------|-------------|------------|
| **Database** | SQLite | PostgreSQL, CockroachDB |
| **File Store** | Local filesystem | S3, GCS, MinIO |
| **Search** | SQLite FTS5 | Hybrid vector + text (embeddings) |
| **HTTP** | Single Axum instance | Multiple instances + load balancer |
| **Auth** | None (localhost) | Tokens → OIDC/SAML |

Three traits define the abstraction:

- **`ResourceStore`** — metadata CRUD operations (get, list, publish, search)
- **`BlobStore`** — file storage (store, fetch, delete, exists)
- **`SearchBackend`** — search indexing and querying

## Planned Enterprise Features

| Feature | Description |
|---------|-------------|
| **Audit Logging** | Track who published what and when. `audit_log` table with actor, action, and context. |
| **Webhooks** | Notify external systems on `resource.published` and `resource.deleted` events. HMAC-signed payloads with retry. |
| **Semantic Search** | Hybrid vector + text search via embeddings generated on publish. |
| **Offline Bundles** | `relava export` / `relava import-bundle` for air-gapped environments. Portable tar.gz archives with transitive dependencies. |
| **API Versioning** | `/api/v1` supported indefinitely. Breaking changes ship as `/api/v2` with 6-month deprecation on v1. |
